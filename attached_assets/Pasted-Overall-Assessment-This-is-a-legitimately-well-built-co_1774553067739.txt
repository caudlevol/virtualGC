Quote Accuracy: Where It Breaks Down
This is the most important section. The quotes will be directionally correct but structurally flawed in three specific ways right now.
Problem 1: The pricing engine ignores the line item description entirely.
In costLookup.ts, when the AI generates a scope item like "Cabinet replacement — 30 linear feet" under the kitchen category, the engine does not read "cabinet replacement." It fetches the first matching material row for the kitchen category at the requested quality tier. That means a kitchen backsplash and a full cabinet replacement both get priced at the same per-unit material cost — whatever row happens to be first in the DB. This is the single biggest accuracy gap in the entire codebase.
The fix: The material lookup needs to match on description keywords, not just category. A simple keyword map (cabinet → cabinets, countertop → countertop, backsplash → backsplash) would dramatically improve accuracy. Alternatively, pass the description to the DB query and use a fuzzy match.
Problem 2: The labor formula is a flat divisor that does not reflect real-world billing.
The labor cost per unit is calculated as (hourlyRate × productivityFactor) / 4. That divisor of 4 is arbitrary — it was likely chosen to produce reasonable-looking numbers, but it has no real-world basis. A plumber at $55/hour does not produce 4 units of output per hour for every type of task. A toilet installation (each) and pipe replacement (linear_ft) have completely different labor ratios. The formula treats them identically.
The fix: Add a laborHoursPerUnit field to the laborRatesTable schema, keyed by tradeType + unit. For example: plumbing + each = 2.5 hours, plumbing + linear_ft = 0.25 hours. This is a one-time data enrichment that would make labor costs meaningfully more accurate.
Problem 3: The demo route uses a completely different (and less accurate) pricing path.
In demo.ts, the estimate is generated by calling generateDemoEstimate() and then simply multiplying the AI's own materialCost and laborCost values by the regional factor. It bypasses priceLineItemsFromCostEngine entirely. This means the demo — which is your primary acquisition tool — shows numbers that are pure GPT-4o hallucination adjusted by a zip code multiplier, not numbers grounded in your cost engine. If an agent demos the tool and the numbers look off, they will not subscribe.
The fix: Route the demo through the same priceLineItemsFromCostEngine path as authenticated quotes. The demo should be the most accurate version of the product, not the least.
Upgrade Opportunities (Prioritized)
These are ranked by impact-to-effort ratio for your 2026 timeline.
1. Description-Aware Material Matching (High Impact / Medium Effort)
Fix the costLookup.ts to match on description keywords before falling back to category. This single change closes the largest accuracy gap and makes the Claude review flags meaningful rather than noise.
2. Unify Demo and Authenticated Pricing Paths (High Impact / Low Effort)
One-line change in demo.ts to call priceLineItemsFromCostEngine instead of using raw AI costs. This makes your demo trustworthy, which directly impacts conversion.
3. Add laborHoursPerUnit to the Labor Rate Schema (Medium Impact / Medium Effort)
Enrich the seed data with realistic hours-per-unit by trade and unit type. This is a data problem, not a code problem — the schema change is small, the research is the work.
4. Expand Regional Multipliers Beyond Top 50 Metros (Medium Impact / Low Effort)
Currently, any zip code outside the 50 seeded metros defaults to a 1.0x national average multiplier. That means a quote in rural Alabama and a quote in suburban Connecticut both get the same multiplier. The BLS QCEW live integration is already built — it just needs to be triggered more aggressively for unknown zip prefixes rather than defaulting to 1.0.
5. Add a "Year Built" Aging Factor to the Cost Engine (Medium Impact / Low Effort)
The property's yearBuilt is captured and passed to the AI context, but the cost engine never uses it. A home built in 1955 has dramatically different renovation costs than one built in 2005 — knob-and-tube wiring, cast iron pipes, plaster walls, asbestos risk. Adding a simple age multiplier (e.g., pre-1970 = 1.25x labor, pre-1990 = 1.10x) would make quotes for older homes significantly more credible.
6. Shareable Quote Co-Branding (High Impact for Revenue / Low Effort)
The shared quote route already fetches agentName and orgName from the DB. The frontend just needs a header on the shared quote page that says "Prepared by [Agent Name] via WalkToQuote" with a logo slot. This is the white-label seed that unlocks brokerage-tier pricing — and it is one UI component away.
